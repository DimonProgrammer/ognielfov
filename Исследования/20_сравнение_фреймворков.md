# Framework Comparison: Corporate Merch Website "Огни Эльфов"

## Project Context

- **Site:** elfprint.ru — corporate merch production, Moscow
- **Scope:** ~15-20 pages (home, catalog 6-8 categories, services/print tech, blog 5-10 articles, about, contacts, portfolio, legal pages)
- **Design:** Apple-style minimalism, Onest font, black/white + lime accent (#C6F24E), Tailwind CSS
- **Current state:** HTML prototype with Tailwind CDN exists
- **Deployment target:** Russian VPS (not Vercel/Netlify)
- **Forms:** Send to Telegram Bot API
- **SEO:** Critical — Yandex + Google
- **Developer tools:** Claude Code as primary, Cursor as secondary

## 1. NEXT.JS 15 (App Router)

### Without Vercel — What's Lost

| Feature | On Vercel | On VPS (self-hosted) | Impact |
|---------|-----------|---------------------|--------|
| Edge Runtime | Native | Not available (use Node.js runtime) | Low — not needed for SSG site |
| ISR (Incremental Static Regeneration) | Native, on-demand | Works with `next start` but no distributed cache | Medium — can use revalidate timers |
| Image Optimization | Vercel Image CDN | Must run `next start` with Node.js, or use `sharp` | Medium — needs Node.js process |
| Analytics | Vercel Analytics | Must use Yandex.Metrika / GA4 anyway | None — we need Russian analytics |
| Edge Middleware | Full edge network | Runs on Node.js server | Low — barely needed |
| Deployment | `git push` | Docker build + deploy scripts | Medium — one-time setup |
| Preview Deployments | Native | Must configure separately | Low — small team |

**Verdict:** For an SSG-dominant site, loss is minimal. The main inconvenience is that `next/image` requires a running Node.js process even for statically exported sites (or you must use `unoptimized: true` and handle images yourself).

### Static Export (`output: 'export'`)

Next.js 15 supports full static export, BUT:
- **No Image Optimization** (must set `images: { unoptimized: true }`)
- **No Server Components runtime** (they become static at build)
- **No API Routes** — forms must use client-side fetch to external endpoints
- **No Middleware**
- **No ISR/revalidation**

With `generateStaticParams` + `output: 'export'`, you get pure static HTML files that can be served by nginx. This eliminates the need for Node.js on the server entirely.

### SSG Reliability

`generateStaticParams` is mature and reliable in Next.js 15. For a site with ~20 static pages and ~10 blog posts, it works perfectly. No edge cases at this scale.

### Deployment on Russian VPS

**Option A: Static Export + nginx**
```
next build → out/ directory → upload to VPS → serve with nginx
```
Simplicity: 9/10. Just static files.

**Option B: Docker + Node.js**
```dockerfile
FROM node:20-alpine
COPY . .
RUN npm ci && npm run build
EXPOSE 3000
CMD ["npm", "start"]
```
Then nginx reverse proxy to port 3000. Needed if you want ISR or API routes.

**Option C: Docker Compose**
```yaml
services:
  web:
    build: .
    ports: ["3000:3000"]
  nginx:
    image: nginx
    volumes: [./nginx.conf:/etc/nginx/conf.d/default.conf]
    ports: ["80:80", "443:443"]
```

For this project, Option A is sufficient. Complexity: Low.

### shadcn/ui

Native support. shadcn/ui is built for Next.js + React + Tailwind. All components work out of the box. Copy-paste components, full customization, no external dependencies.

Relevant components for this project: Button, Card, Accordion, Dialog, Form, Input, Textarea, Tabs, Badge, Carousel, Sheet (mobile nav).

### SEO Capabilities

- **Metadata API**: `generateMetadata()` for dynamic meta tags per page — excellent
- **Built-in sitemap**: `sitemap.ts` generates XML sitemap
- **robots.ts**: Programmatic robots.txt
- **Schema.org**: Manual via JSON-LD in `<script>` tags (no built-in helper, but straightforward)
- **Open Graph images**: `opengraph-image.tsx` for auto-generated OG images
- **Canonical URLs**: Via metadata API
- **Alternate/hreflang**: Supported (not needed for single-language)
- **HTML output**: Clean semantic HTML, SSG pages are fully rendered

**Yandex-specific considerations:**
- Yandex crawler handles SSG HTML perfectly
- Yandex.Metrika script added via `next/script` or layout
- Turbo Pages: Can generate Turbo feed from blog content
- Yandex.Webmaster integration: Sitemap submission works normally

### Bundle Size

For a content-focused SSG site:
- Next.js runtime: ~85-95 KB (gzipped) for React + Next.js client runtime
- Per-page JS: ~5-15 KB for hydration
- With static export: Still includes React hydration JS
- **Total first-load JS**: ~100-120 KB (gzipped)

This is the heaviest option. React hydration JS ships even for pages that have no interactivity.

### AI Tool Compatibility (Claude Code / Cursor)

**Excellent.** Next.js has the largest training data representation in LLMs:
- Claude Code generates Next.js App Router code reliably
- File-based routing is predictable and AI-friendly
- shadcn/ui components are well-known to AI models
- TypeScript support is first-class
- Massive community = more training data = better AI suggestions
- AI can scaffold entire pages with `page.tsx`, `layout.tsx`, `loading.tsx`

### Development Speed

For ~15 pages:
- Scaffolding: ~1 hour (with Claude Code)
- Component library setup (shadcn/ui): ~30 min
- Individual page development: ~2-4 hours per complex page, ~1 hour per simple page
- Blog setup with MDX: ~2-3 hours (using next-mdx-remote or contentlayer2)
- Form + Telegram integration: ~2-3 hours
- SEO setup: ~1-2 hours
- **Estimated total: 3-4 days**

## 2. ASTRO 5

### SSG by Default

Astro is SSG-first. Every page renders to static HTML at build time with **zero JavaScript by default**. This is the fundamental advantage: pages are pure HTML + CSS until you explicitly add interactive components.

For a corporate website that is 90% static content with only forms and maybe a mobile menu as interactive elements, this philosophy is ideal.

### Islands Architecture for Forms

Interactive "islands" are opt-in via `client:*` directives:

```astro
---
import ContactForm from '../components/ContactForm.tsx';
---
<h1>Контакты</h1>
<p>Статический контент — 0 KB JavaScript</p>

<!-- Only this component ships JS -->
<ContactForm client:visible />
```

Directives:
- `client:load` — hydrate immediately (critical forms)
- `client:visible` — hydrate when scrolled into view (contact form at bottom)
- `client:idle` — hydrate when browser is idle
- `client:media="(max-width: 768px)"` — hydrate conditionally

For this project: Contact form, callback form, mobile menu = 3-4 islands. Everything else is zero JS.

### Astro + React Components

Astro supports React via `@astrojs/react`. You write `.astro` files for pages/layouts and `.tsx` for interactive components. The integration is mature and well-tested.

```
src/
  pages/          # .astro files — SSG, zero JS
  components/
    layout/       # .astro files — Header, Footer (static)
    ui/           # .tsx files — shadcn/ui components (interactive)
    forms/        # .tsx files — ContactForm, CallbackForm
```

### Deployment

**Pure SSG (no SSR):**
```
astro build → dist/ directory → upload anywhere
```
Serve with nginx, Apache, any static file server. No Node.js needed on the server.

This is the simplest deployment possible. Just static files.

**If SSR needed later:**
```
adapter: node → Docker with Node.js
adapter: cloudflare/netlify → those platforms
```

For this project: Pure static is sufficient. SSR not needed.

### shadcn/ui Integration

Via `@astrojs/react` + shadcn/ui. Not as seamless as Next.js but works well.

**Setup process:**
1. Install `@astrojs/react`, `tailwindcss`
2. Initialize shadcn/ui with React
3. Components go in `src/components/ui/`
4. Use in `.astro` files with `client:*` when interactive

**Caveats:**
- shadcn/ui components that are purely visual (Card, Badge) work without `client:*` — they render as static HTML
- Interactive components (Dialog, Accordion, Form) need `client:load` or `client:visible`
- Some components require adjusting import paths
- The shadcn CLI `npx shadcn@latest add` works but may need manual path adjustments

**Quality:** 7/10 — works but requires more manual configuration than Next.js.

### Content Collections

Astro Content Collections (v5 with Content Layer API) is excellent:

```typescript
// src/content.config.ts
import { defineCollection, z } from 'astro:content';

const blog = defineCollection({
  type: 'content', // MDX files
  schema: z.object({
    title: z.string(),
    description: z.string(),
    date: z.date(),
    image: z.string().optional(),
  }),
});

const catalog = defineCollection({
  type: 'data', // JSON/YAML files
  schema: z.object({
    name: z.string(),
    description: z.string(),
    image: z.string(),
    features: z.array(z.string()),
  }),
});
```

**vs Next.js contentlayer2:**
- Content Collections are built-in (no extra dependency)
- Type-safe with Zod schemas
- Better DX for content-heavy sites
- contentlayer2 is a community fork with uncertain maintenance

**Winner: Astro** by a significant margin for content management.

### SEO

- **@astrojs/sitemap**: Auto-generates sitemap.xml from all pages
- **Astro.props for meta**: Full control via `<head>` in layouts
- **RSS feed**: `@astrojs/rss` for blog
- **Schema.org**: Manual JSON-LD (same as Next.js)
- **HTML output**: The cleanest HTML of all options — no framework artifacts
- **Zero JS pages**: Yandex crawler sees pure HTML — ideal for indexing
- **View Transitions**: Native API, no JS framework needed

**Yandex-specific:** Pure static HTML is the gold standard for Yandex indexing. No hydration artifacts, no client-side rendering concerns. Yandex.Metrika integrates via a simple `<script>` tag in the layout.

### Bundle Size

- **Static pages: 0 KB JavaScript** (unless you add islands)
- **Page with contact form island: ~45-60 KB** (React + form component, gzipped)
- **Average across site: ~5-10 KB per page** (most pages are zero-JS)

**This is the lightest option by far.** A catalog page that shows products as cards? 0 KB JS. A blog article? 0 KB JS. Only the contact page and pages with interactive elements ship any JavaScript.

### AI Tool Compatibility

**Good, but not as strong as Next.js.**
- `.astro` file syntax is specific to Astro — LLMs have less training data
- Claude Code handles Astro well but occasionally confuses syntax with JSX
- React component files (.tsx) work perfectly with AI tools
- Content Collections schema writing is straightforward for AI
- File-based routing is AI-friendly
- Growing ecosystem = improving AI support

**Rating: 7.5/10** (vs Next.js 9/10)

### Development Speed

For ~15 pages:
- Scaffolding: ~1 hour
- Tailwind + shadcn/ui setup: ~1 hour (more config needed)
- Individual pages: ~1.5-3 hours per complex page, ~30 min per simple page (`.astro` templates are faster than React for static content)
- Blog setup with Content Collections: ~1.5 hours (built-in, faster than Next.js)
- Catalog with Content Collections: ~1 hour
- Form + Telegram integration: ~2-3 hours
- SEO setup: ~1 hour
- **Estimated total: 2.5-3.5 days**

Faster than Next.js for content-heavy static sites because `.astro` files are simpler than React components for static content.

## 3. NUXT 4

### SSG Mode (`nuxt generate`)

Nuxt's `nuxt generate` pre-renders all pages to static HTML. Works well for the project scope.

**Configuration:**
```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  nitro: {
    prerender: {
      routes: ['/blog/article-1', '/blog/article-2', ...],
      crawlLinks: true,
    }
  }
})
```

**Quality:** Reliable but less refined than Astro for pure SSG. Nuxt is primarily designed for SSR with SSG as a secondary mode. The `crawlLinks` approach can miss pages if they're not linked from the home page.

### Vue vs React Ecosystem

For this type of corporate website:

| Aspect | Vue/Nuxt | React/Next.js |
|--------|----------|---------------|
| Template syntax | More intuitive for content pages | JSX more powerful but verbose |
| Component libraries | Nuxt UI, Vuetify, PrimeVue | shadcn/ui, Radix, headless UI |
| Tailwind integration | Native in Nuxt | Native in Next.js |
| TypeScript | Good (improved in Vue 3) | Excellent |
| AI training data | Less | Significantly more |
| Russian dev community | Strong (popular in RU) | Strong |

Vue's template syntax is cleaner for content-heavy pages, but the React ecosystem is larger for component libraries.

### Deployment on Russian VPS

Similar to Next.js:
- **Static (nuxt generate):** Upload `.output/public/` to nginx. Simple.
- **SSR:** Docker + Node.js + nginx reverse proxy.

### SEO

- **@nuxtjs/seo**: Comprehensive SEO module (sitemap, robots, schema.org, OG images)
- **useHead()**: Composable for meta tags
- **useSeoMeta()**: Dedicated SEO meta composable
- **Nuxt SEO module** is arguably the best all-in-one SEO solution across all frameworks

**Quality: 9/10** — Nuxt SEO module is excellent and more batteries-included than Next.js or Astro.

### Component Libraries (shadcn/ui equivalent)

- **Nuxt UI (by NuxtLabs)**: The closest equivalent. Tailwind-based, highly customizable. Good quality but smaller ecosystem than shadcn/ui.
- **shadcn-vue**: Community port of shadcn/ui for Vue. Works but less mature.
- **PrimeVue**: Full-featured but heavier and different design philosophy.
- **Radix Vue**: Headless primitives (like Radix for React).

**Best option: Nuxt UI** — native integration, Tailwind-based, good quality. But shadcn/ui for React is more polished.

### AI Tool Compatibility

**Moderate.**
- Vue/Nuxt has less representation in LLM training data than React/Next.js
- Claude Code handles Vue well but not as fluently as React
- Nuxt-specific APIs (composables, auto-imports) can trip up AI tools
- Nuxt UI components less known to AI than shadcn/ui
- Vue SFC (Single File Components) syntax is well-understood

**Rating: 6.5/10**

### Development Speed

For ~15 pages:
- Scaffolding: ~1 hour
- Nuxt UI setup: ~30 min (built-in)
- Pages: ~2-3 hours per complex, ~1 hour per simple
- Blog: ~2 hours (Nuxt Content module)
- Forms: ~2-3 hours
- SEO: ~1 hour (Nuxt SEO module covers a lot automatically)
- **Estimated total: 3-4 days**

## 4. 11TY (ELEVENTY)

### Pure Static Generator

11ty is the simplest option. No JavaScript framework, no virtual DOM, no hydration. Just templates that compile to HTML.

### Maximum Simplicity

```
src/
  _includes/
    layout.njk       # Base layout with Nunjucks
    header.njk
    footer.njk
  pages/
    index.njk
    about.njk
    contacts.njk
  blog/
    post-1.md         # Markdown with front matter
    post-2.md
  catalog/
    category-1.njk
  _data/
    catalog.json      # Catalog data
    site.json         # Site-wide data
```

### No JavaScript Framework

- Templates: Nunjucks, Liquid, EJS, or plain HTML
- Styling: Tailwind CSS via PostCSS pipeline
- Interactivity: Vanilla JS or Alpine.js for forms/menu
- Components: Nunjucks partials/macros (not React-style components)

### Deployment

The absolute simplest:
```
npx @11ty/eleventy → _site/ → upload to literally any web server
```
Works on any hosting that can serve HTML files. Shared hosting, VPS, CDN, S3, whatever.

### Limitations for This Project

- **No component model**: No React/Vue components. Nunjucks macros are not the same.
- **No shadcn/ui**: Must build all UI components from scratch with Tailwind
- **Forms**: Alpine.js + fetch() for Telegram integration — more manual work
- **Blog**: Works great with Markdown + front matter
- **Catalog**: Works with JSON data + Nunjucks templates — but no type safety
- **Design system**: Harder to maintain without a component model
- **Interactive elements**: Limited to Alpine.js or vanilla JS

### SEO

- **Full control**: You write the HTML, so you control everything
- **Sitemap plugin**: `@11ty/eleventy-navigation` + custom sitemap template
- **RSS**: `@11ty/eleventy-plugin-rss`
- **Clean HTML**: The cleanest possible output (like Astro but without any framework artifacts)

### AI Tool Compatibility

**Low-Moderate.**
- 11ty has limited representation in LLM training data
- Nunjucks template syntax is less common than JSX or Vue SFC
- Configuration is JavaScript-based (`.eleventy.js`) which AI handles OK
- Claude Code can write Nunjucks but with lower confidence
- No type safety = AI can introduce bugs that TypeScript would catch

**Rating: 5/10**

### Bundle Size

- **0 KB JavaScript** (unless you add Alpine.js or vanilla JS)
- With Alpine.js for forms: ~15 KB gzipped
- **The absolute lightest option**

### Development Speed

For ~15 pages:
- Scaffolding: ~1.5 hours (more boilerplate for layouts)
- Tailwind setup: ~30 min
- UI components (manual): ~4-6 hours (no component library, build from scratch)
- Pages: ~2-3 hours per complex, ~45 min per simple
- Blog: ~1.5 hours (simple Markdown)
- Forms: ~3-4 hours (Alpine.js + vanilla JS, more manual)
- SEO: ~1.5 hours
- **Estimated total: 4-5 days**

Slower because you build everything from scratch. No component library.

## DETAILED RATIONALE FOR SCORES

### Why Astro wins SEO (9) over Next.js (8):
- Astro outputs pure HTML with zero JS on content pages — Yandex crawler sees exactly what users see
- Next.js always ships React hydration JS (~100KB), which doesn't help SEO and can slow TTI
- Both have good meta/sitemap tooling, but Astro's HTML output is cleaner

### Why Astro wins Deployment (10) over Next.js (7):
- Astro SSG: `dist/` folder, upload to nginx. Done. No Node.js needed.
- Next.js static export: Works but `next/image` is crippled without Node.js. Must use `unoptimized: true`.
- Next.js with ISR/Server Components: Needs Node.js process on VPS = more complexity.

### Why Next.js wins AI compatibility (9) over Astro (7.5):
- React/Next.js has 3-5x more representation in LLM training data
- Claude Code generates Next.js App Router code nearly perfectly
- `.astro` file syntax is unique and less represented in training data
- shadcn/ui component generation is native for Next.js

### Why Astro wins Development Speed (9) over Next.js (7):
- `.astro` files are simpler than React components for static content
- Content Collections are built-in vs needing contentlayer2
- No React boilerplate for pages that are 100% static
- Less configuration for the SSG use case

### Why 11ty scores low overall despite deployment/performance:
- No component library = building everything from scratch
- Poor AI tool support = slower development
- No TypeScript type safety in templates
- Less scalable if requirements change

## RECOMMENDATION

### Primary: Astro 5

**Astro 5 is the clear winner for this specific project.** Here's why:

1. **The site is 90% static content.** Astro is purpose-built for this. Zero JS where you don't need it, React islands where you do (forms, mobile menu).

2. **Russian VPS deployment is trivial.** Build → upload static files → nginx. No Node.js process to manage, no Docker required (though you can use it for CI/CD).

3. **SEO is maximized.** Pure HTML output is the gold standard for Yandex indexing. No hydration artifacts, no client-side rendering concerns.

4. **Content Collections are excellent** for the blog and catalog — built-in, type-safe, better than any Next.js content solution.

5. **Performance is unbeatable.** 0 KB JS on content pages → perfect Core Web Vitals → better Yandex ranking.

6. **You still get React when needed.** shadcn/ui components work for forms and interactive elements via `@astrojs/react`.

7. **Development speed is fastest** for this type of content-heavy site.

### Mitigating Astro's Weakness (AI Tools)

The main gap is AI tool compatibility (7.5 vs Next.js's 9). Mitigation strategies:

1. **Write interactive components as standard React `.tsx` files** — Claude Code handles these perfectly
2. **Use `.astro` files only for page layouts/templates** — simpler syntax, less room for AI errors
3. **Keep a reference `.astro` file in context** for Claude Code to learn the pattern
4. **shadcn/ui components are standard React** — AI generates them perfectly regardless of framework

This gap is narrowing rapidly as Astro grows in popularity and training data.

### Deployment on Russian VPS

```bash
# Build
npm run build

# Result: dist/ folder with pure static HTML/CSS/JS
# Deploy: rsync to nginx web root
rsync -avz dist/ user@vps:/var/www/elfprint.ru/

# nginx config
server {
    listen 80;
    server_name elfprint.ru www.elfprint.ru;
    root /var/www/elfprint.ru;
    index index.html;

    # SPA-style routing for clean URLs
    location / {
        try_files $uri $uri/ $uri.html =404;
    }

    # Cache static assets
    location ~* \.(js|css|png|jpg|jpeg|webp|svg|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Security headers
    add_header X-Frame-Options DENY;
    add_header X-Content-Type-Options nosniff;
}
```

### Form → Telegram Integration

Since Astro SSG has no server-side API routes, two options:

**Option 1: Client-side fetch to Telegram Bot API directly**
```typescript
// src/components/forms/telegram.ts
const TELEGRAM_BOT_TOKEN = import.meta.env.PUBLIC_TELEGRAM_BOT_TOKEN;
const TELEGRAM_CHAT_ID = import.meta.env.PUBLIC_TELEGRAM_CHAT_ID;

export async function sendToTelegram(message: string) {
  const response = await fetch(
    `https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage`,
    {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        chat_id: TELEGRAM_CHAT_ID,
        text: message,
        parse_mode: 'HTML',
      }),
    }
  );
  return response.json();
}
```

**Caveat:** Bot token exposed in client JS. For a corporate site with no sensitive data, this is acceptable but not ideal.

**Option 2 (Recommended): Lightweight proxy on VPS**
A simple 20-line Node.js/Python script or nginx proxy that forwards form data to Telegram without exposing the bot token:

```python
# /opt/elfprint-forms/server.py — FastAPI, 20 lines
from fastapi import FastAPI
import httpx, os

app = FastAPI()
BOT_TOKEN = os.environ["TELEGRAM_BOT_TOKEN"]
CHAT_ID = os.environ["TELEGRAM_CHAT_ID"]

@app.post("/api/contact")
async def contact(data: dict):
    msg = f"<b>Заявка с сайта</b>\nИмя: {data['name']}\nТел: {data['phone']}\nСообщение: {data.get('message', '—')}"
    async with httpx.AsyncClient() as client:
        await client.post(f"https://api.telegram.org/bot{BOT_TOKEN}/sendMessage",
                         json={"chat_id": CHAT_ID, "text": msg, "parse_mode": "HTML"})
    return {"ok": True}
```

Run with `uvicorn server:app --port 8001`, proxy via nginx at `/api/`.

## ALTERNATIVE RECOMMENDATION: When to Choose Next.js Instead

Choose Next.js 15 if ANY of these become true:
- Client requests a CMS with preview mode (Next.js Draft Mode is superior)
- E-commerce features are added later (payment, cart, user accounts)
- The site needs server-side API routes beyond simple form proxying
- The team grows and standardizing on React/Next.js reduces onboarding friction
- Real-time features are needed (chat, live updates)

For the current scope (static corporate merch website), these scenarios are unlikely. Astro is the right tool.
