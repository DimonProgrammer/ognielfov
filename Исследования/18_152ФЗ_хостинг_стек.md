**Purpose:** Determine legal hosting requirements, best Russian hosting provider, optimal tech stack, and CMS strategy given 152-FZ constraints.

## 1. RUSSIAN DATA LOCALIZATION LAWS (152-FZ)

### 1.1 What Does 152-FZ Require?

Federal Law No. 152-FZ "On Personal Data" (enacted 2006, significantly amended in 2024-2025) is Russia's primary data protection law.

**Core localization requirement (Article 18, Part 5 -- new wording effective July 1, 2025):**
- All **primary collection, recording, systematization, accumulation, and storage** of personal data of Russian citizens must be conducted **exclusively on servers physically located in the Russian Federation**.
- The July 2025 amendment changed the wording from a positive obligation ("you must use Russian databases") to an **explicit prohibition** ("it is forbidden to use databases outside Russia for primary processing").
- Cross-border transfer is still possible **after** data has been first collected and stored in Russia, and only with proper legal basis and Roskomnadzor notification.

**Key 2025-2026 changes:**
- **September 1, 2025:** Consent for data processing must be a **separate standalone document**, not a clause in user agreements or privacy policies.
- **May 30, 2025:** Dramatically increased fines took effect (Law No. 420-FZ of November 2024).
- Cookies containing identifying data are now explicitly treated as personal data by Roskomnadzor.
- Google Analytics, Mailchimp, Notion, and similar foreign SaaS services that process personal data are effectively **prohibited** unless data is first stored in Russia.

### 1.2 Does a Lead Collection Website Fall Under 152-FZ?

**YES, unequivocally.**

Even the simplest feedback form on a website collecting a name + phone number or email makes the organization a **personal data operator** (оператор персональных данных) with the **full list of obligations**:

- Register as a data operator with Roskomnadzor (via notification)
- Publish a Personal Data Processing Policy on the website
- Obtain explicit user consent via checkbox under each form
- Specify processing purpose, storage period, protection measures
- Ensure data protection per law requirements
- Store data on Russian servers

**There are NO exemptions** for "just contact data." The only narrow exemption is for data consisting of *only* surnames, first names, and patronymics -- but since the site also collects phone/email, this exemption does not apply. The site would also likely store data about clients who contacted but haven't concluded a contract, further negating any exemptions.

### 1.3 Is Vercel Acceptable for Hosting?

**NO. Vercel is NOT acceptable** for a Russian B2B site that collects personal data.

Reasons:
1. **Legal:** Vercel hosts on AWS infrastructure with data centers primarily in the US. Since July 2025, it is **explicitly prohibited** to collect personal data of Russian citizens into databases located outside Russia. Even if the website itself is static, form submissions that go through Vercel serverless functions would violate 152-FZ.
2. **Terms of Service:** Vercel's "Hobby" tier is for personal, non-commercial projects only. Using it for a commercial site violates their ToS.
3. **No Russian region:** Vercel does not offer data center regions in Russia.

**The only legal architecture with Vercel** would be: host the static frontend on Vercel BUT send all form data directly to a backend API hosted on Russian servers. However, even this is risky because:
- Cookies and analytics data would still be processed abroad
- Roskomnadzor may interpret the form submission journey as "primary collection" happening on foreign servers
- It adds unnecessary architectural complexity

**Recommendation:** Use Russian hosting entirely. It eliminates all legal risk.

### 1.4 Penalties for Non-Compliance (as of May 30, 2025)

| Violation Type | First Offense (Legal Entity) | Repeat Offense |
|---|---|---|
| Processing without legal basis | 150,000 - 300,000 RUB | 300,000 - 500,000 RUB |
| Processing without consent | 300,000 - 700,000 RUB | 1,000,000 - 1,500,000 RUB |
| Failure to publish privacy policy | 60,000 - 100,000 RUB | 100,000 - 300,000 RUB |
| Data breach (1K-10K records) | 3,000,000 - 5,000,000 RUB | Revenue-based |
| Data breach (10K-100K records) | 5,000,000 - 10,000,000 RUB | Revenue-based |
| Data breach (100K+ records) | 10,000,000 - 15,000,000 RUB | Revenue-based |
| Biometric data breach | 15,000,000 - 20,000,000 RUB | Revenue-based |
| **Revenue-based fines (repeat)** | **1-3% of annual revenue** | Min 20M, max 500M RUB |

These fines are serious and enforceable. Roskomnadzor has the authority to initiate administrative proceedings and can block non-compliant websites.

### 1.5 Required Compliance Measures for elfprint.ru

1. Host on Russian servers (data center in Moscow preferred for speed)
2. Publish privacy policy (Политика обработки персональных данных) with link in footer
3. Add consent checkbox under every form: "Согласен на обработку персональных данных"
4. Register as data operator with Roskomnadzor
5. Use Yandex.Metrica instead of Google Analytics
6. Use Russian email services (or self-hosted) for form notifications
7. Implement cookie consent banner with category separation
8. Store form submissions in a database on Russian servers

## 3. IMPACT ON TECH STACK

### 3.1 Can Next.js App Router Work on Russian Hosting?

**Yes, but with caveats.**

Next.js can be self-hosted via:
1. **Node.js server** (`next start`) -- supports all features including Server Components, Server Actions, streaming
2. **Docker container** -- using `output: "standalone"` in next.config.js, creates a minimal server without node_modules
3. **Static export** -- `output: "export"` generates pure HTML/CSS/JS, but loses SSR, Server Actions, middleware, image optimization

**Self-hosting considerations:**
- Need to set up reverse proxy (Nginx) manually
- Image optimization requires `sharp` package installation
- No automatic ISR/caching without custom setup
- Streaming requires Nginx to disable buffering
- No Edge Runtime (Vercel-specific)
- No Analytics/Speed Insights (Vercel-specific)
- Need to manage process lifecycle (PM2 or systemd)

**Verdict:** Next.js works on Russian VPS but requires more DevOps effort compared to Vercel. The DX gap is manageable for a developer comfortable with Docker.

### 3.2 Would Astro Be Simpler?

**Yes, significantly simpler for this project type.**

Astro advantages for Russian hosting:
- **Default static output:** Generates plain HTML/CSS/JS files -- can be served by any web server (Nginx, Caddy, Apache)
- **Zero JavaScript by default:** Only interactive "islands" send JS, resulting in 2-3x faster page loads vs Next.js
- **Build speed:** ~3x faster than Next.js (18s vs 52s for 1000 pages)
- **Deployment simplicity:** Just upload dist/ folder to any hosting; no Node.js runtime needed for static mode
- **Server Islands (Astro 5.0):** Can defer dynamic content rendering while keeping the site mostly static
- **Docker:** Simple multi-stage build: build with Node.js, serve with Nginx (total image ~30MB)
- **Framework agnostic:** Can use React, Vue, Svelte, or plain HTML for components
- **Cheaper hosting:** Static files = cheaper hosting tier, no Node.js process running 24/7

**Astro Docker deployment (static):**
```
Stage 1: node:20-alpine -> npm ci && npm run build
Stage 2: nginx:alpine -> copy dist/ to /usr/share/nginx/html
```

**Astro Docker deployment (SSR with Node.js adapter):**
```
Stage 1: node:20-alpine -> npm ci && npm run build
Stage 2: node:20-alpine -> copy dist/ -> node server.js
```

**For elfprint.ru specifically:**
- The site is primarily content/marketing (service pages, portfolio, about, blog)
- Forms can submit to a simple API endpoint (could even be a serverless function or separate Node.js microservice)
- Interactive elements (calculators, portfolio galleries) work as Astro islands
- Blog/content pages are perfect for static generation

### 3.3 Nuxt.js as an Option?

**Viable but not the best fit.**

Nuxt.js (Vue-based) advantages:
- **Nitro server engine:** Adapters for Node, Deno, Cloudflare Workers, Docker -- same build deploys anywhere with zero code change
- **Flexible rendering:** SSR, SSG, ISG, hybrid (per-route) out of the box
- **Self-hosting friendly:** Less Vercel-dependent than Next.js
- **Predictable costs:** Fixed Node.js server vs Vercel's usage-based pricing

Drawbacks for this project:
- Requires Node.js runtime (unlike Astro static)
- Vue ecosystem is smaller than React (fewer component libraries)
- Team expertise: if the developer knows React, switching to Vue adds learning curve
- For a content-heavy marketing site, it's more framework than needed

**Note (July 2025):** Vercel acquired NuxtLabs, but Nuxt remains MIT-licensed and open source.

### 3.4 Pure SSG with Astro -- THE RECOMMENDED APPROACH

**This is the optimal architecture for elfprint.ru:**

```
Architecture:
├── Frontend: Astro (static SSG)
│   ├── Pages: HTML/CSS, zero JS
│   ├── Interactive: React/Svelte islands (calculator, gallery, form)
│   └── Deploy: Nginx on Timeweb VPS (Moscow)
│
├── Forms API: Lightweight Node.js service OR Telegram Bot API
│   ├── Receives form submissions
│   ├── Stores in PostgreSQL on same VPS
│   ├── Sends notifications (Telegram, email)
│   └── Deploy: Same VPS, Docker
│
└── Analytics: Yandex.Metrica (free, 152-FZ compliant)
```

**Why this is best:**
1. **Simplest deployment:** Static files on Nginx = maximum reliability, minimal maintenance
2. **Fastest performance:** Pre-built HTML, CDN-ready, 0.5s first paint
3. **Cheapest hosting:** Static files use minimal resources; even Cloud-15 (450 RUB/mo) is sufficient
4. **152-FZ compliant:** Everything on Russian servers, no foreign services
5. **SEO optimal:** Perfect Lighthouse scores out of the box
6. **Developer experience:** Astro is simple to learn, great docs, supports React components
7. **Future-proof:** Can add SSR islands later if needed (Astro 5.0 Server Islands)
8. **CI/CD:** GitHub Actions -> build -> scp/rsync to VPS (simple, no K8s needed)

## 4. CMS CONSIDERATIONS

### 4.1 Is Sanity Acceptable Under 152-FZ?

**NO, not by default.**

Sanity stores content data on their cloud (US/EU servers). If the content database contains any personal data (author names, contact info, user-generated content), it violates 152-FZ.

However, for a **purely content-management** use case (managing page text, images, blog posts) where no personal data is stored in Sanity, it could technically be used as a content source. But:
- Legal risk remains if Roskomnadzor interprets broadly
- If CMS user accounts (editors) are Russian citizens, their login data is personal data
- Adds dependency on a US service (geopolitical risk, potential sanctions-related blocks)

**Recommendation:** Avoid Sanity. Use self-hosted CMS on Russian servers.

### 4.2 Self-Hosted Headless CMS Options

#### Payload CMS -- BEST FIT for Next.js/Astro stack
- **Architecture:** Next.js native, installs directly in /app folder
- **Self-hosting:** Docker on any VPS, needs Node.js + PostgreSQL
- **Requirements:** Min 1GB RAM
- **Strengths:** TypeScript-first, built for developers, admin panel included, REST + GraphQL API, open source (MIT)
- **Weakness:** Younger ecosystem, designed for persistent Node.js (not serverless)
- **152-FZ:** Fully compliant when self-hosted on Russian VPS

#### Strapi
- **Architecture:** Node.js + React admin panel
- **Self-hosting:** Docker, needs Node.js + PostgreSQL/MySQL/SQLite
- **Strengths:** Largest plugin marketplace, mature ecosystem, visual content type builder, REST + GraphQL, role-based access, i18n
- **Weakness:** Enterprise features (versioning, SSO) behind paywall; can be resource-intensive
- **152-FZ:** Fully compliant when self-hosted

#### Directus
- **Architecture:** Database-first (wraps existing SQL database), Vue.js admin
- **Self-hosting:** Docker, supports PostgreSQL, MySQL, SQLite, MSSQL, Oracle
- **Strengths:** Works with existing databases, intuitive UI for non-technical editors, automation features, real-time updates, no vendor lock-in
- **Weakness:** Less developer-centric than Payload/Strapi
- **152-FZ:** Fully compliant when self-hosted

#### 1C-Bitrix
- **Architecture:** Monolithic PHP CMS, NOT truly headless
- **REST API:** Has API capabilities but designed as traditional CMS
- **Strengths:** Dominant in Russian market, huge ecosystem, integrations with 1C (accounting), Russian-language documentation and support
- **Weakness:** Heavy, PHP-based, not designed for modern JS frontends, expensive licensing, poor developer experience for React/Vue integration
- **152-FZ:** Fully compliant (Russian company, Russian servers)
- **Recommendation:** Avoid for this project. It's the opposite of modern headless architecture.

### 4.3 CMS Recommendation for elfprint.ru

**Phase 1 (MVP/Launch): No CMS needed.**
- Content changes are infrequent for a corporate site
- Use Astro's built-in Markdown/MDX for blog posts
- Developer (you) can update content via git commits
- This is the simplest, fastest, most reliable approach

**Phase 2 (Post-launch, if content team needs autonomy): Directus**
- Best for non-technical content editors (the client's team)
- Database-first approach = easy to set up alongside existing PostgreSQL
- Docker deployment on the same Timeweb VPS
- Intuitive Vue.js admin panel
- REST/GraphQL API to feed Astro at build time (SSG) or runtime (SSR islands)
- Lightweight enough to run on the same VPS

**Alternative Phase 2: Payload CMS** if the developer wants tighter integration with the JS ecosystem and TypeScript-first approach.

```
│              RECOMMENDED STACK               │
├─────────────────────────────────────────────┤
│                                             │

│  Forms:        Custom Node.js API endpoint  │
│  Database:     PostgreSQL (form data only)  │
│  CMS:          None (Phase 1) /             │
│                Directus (Phase 2)           │
│  Analytics:    Yandex.Metrica               │
│  Hosting:      Timeweb Cloud VPS (Moscow)   │
│  Web Server:   Nginx                        │
│  Container:    Docker                        │
│  CI/CD:        GitHub Actions -> VPS deploy  │
│  SSL:          Let's Encrypt (free)         │
│  Notifications: Telegram Bot API            │
│                                             │
│  Est. Cost:    ~500-900 RUB/mo (~$5-9)     │
│  152-FZ:       Fully compliant              │
│                                             │
└─────────────────────────────────────────────┘
```

### Why This Stack Wins:

1. **Legal compliance:** Everything on Russian servers, no foreign data processing
2. **Performance:** Static HTML = fastest possible load times, perfect for SEO
3. **Cost:** Under 1,000 RUB/month for hosting
4. **Simplicity:** No complex infrastructure, no Kubernetes, no container orchestration
5. **Reliability:** Static files + Nginx = virtually zero downtime
6. **Developer experience:** Astro is fast to develop, supports React components
7. **Scalability:** Can add CDN, upgrade VPS, or add Directus CMS later
8. **Maintenance:** Minimal -- static files don't crash, no runtime errors

### Key Trade-offs vs. Vercel/Next.js:
- Lose: One-click deploy, Edge network, automatic ISR, Preview deployments, Analytics dashboard
- Gain: Legal compliance, lower cost, full control, independence from US infrastructure, predictable billing
